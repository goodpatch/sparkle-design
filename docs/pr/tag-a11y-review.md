## 対応内容（修正が必要な場合）

本レビューはレポート作成のみであり、コード修正は実施していません。以下は推奨される対応事項です。

### Fail 項目

Tag コンポーネントには Fail 判定の項目はありません。

### Needs review 項目

#### 1) 1.1.1 非テキストコンテンツ — アイコンのみ使用時の代替テキスト

- **状況**: Tag は `children` に任意の `React.ReactNode` を受け付ける（`index.tsx:145`）。テキストが渡される限り問題ないが、アイコンのみが渡された場合に代替テキストが欠如する可能性がある
- **判断**: Tag の設計上テキストラベルが前提であるため、通常利用では問題なし。アイコンのみでの利用パターンがある場合は `aria-label` の併用をドキュメントで推奨

#### 2) 4.1.2 名前・役割・値 — デフォルト role の不在

- **状況**: Tag は `<div>` でレンダリングされ、デフォルトで `role` が設定されていない
- **判断**: Tag は汎用ラベリングコンポーネントであり、用途が多様（分類タグ / ステータスタグ）。`role` の一律付与は不適切なケースがあるため、利用者が用途に応じて明示的に付与する設計は妥当。ドキュメントでの案内を推奨
  - 分類用途: `role` 不要（プレーンテキストとして読み上げ可能）
  - ステータス用途: `role="status"` を推奨

#### 3) 4.1.3 状態メッセージ — 動的変更時の支援技術通知

- **状況**: Tag の内容が動的に変化した場合、支援技術に通知されない
- **判断**: Tag は静的ラベルが主な用途であり、Badge のようなリアルタイム通知コンポーネントとは異なる。デフォルトの `aria-live` 付与は過剰。動的コンテキストでの利用時に `role="status"` または `aria-live="polite"` の付与をドキュメントで推奨

## 補足

### 総合評価

Tag コンポーネントは情報表示（ラベリング）に特化した非インタラクティブコンポーネントであり、多くのWCAG項目が N/A となる。Fail 項目はなく、全体的にアクセシビリティ上の大きなリスクはない。

Needs review の 3 項目はいずれも「利用コンテキスト依存」であり、コンポーネント自体の修正ではなくドキュメント整備で対応可能。

### Badge コンポーネントとの比較

Tag と Badge は類似した表示コンポーネントだが、以下の点で異なる:

| 観点 | Tag | Badge |
|---|---|---|
| 主な用途 | 分類ラベル / ステータス表示 | 通知カウント表示 |
| テキスト可視性 | 常に可視 | `xs`/`sm` で非表示のケースあり |
| `role` のデフォルト必要性 | 低（汎用ラベル） | 高（通知カウントは `role="status"` が妥当） |
| 動的変化の頻度 | 低（静的ラベル） | 高（カウント増減） |

この違いにより、Tag は Badge ほど厳格な `role`/`aria-live` のデフォルト付与を必要としない。

### 推奨アクション（優先度順）

1. **中**: ドキュメント/Storybook に Tag の用途別アクセシビリティガイドラインを追加
   - 分類用途: そのままで問題なし
   - ステータス用途: `role="status"` の明示を推奨
   - 動的変化する場合: `aria-live="polite"` の追加を推奨
2. **低**: アイコンのみで Tag を使用するケースについて `aria-label` 併用の案内を追加
3. **低**: `tabIndex` 付与時のフォーカスリング未定義について注記を追加
